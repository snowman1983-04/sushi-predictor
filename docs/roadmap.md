# v2.0 ロードマップ：実務向け製造判断支援ツールへの進化案

本ロードマップは、現状のv1.1（学習用ポートフォリオ）から、実務利用可能な製造判断支援ツールへ進化させるための設計提案です。

---

## 1. 3段階モデルの並列表示

予測値を1つだけ出すのではなく、性質の異なる3種類の予測を並べて提示することで「説明可能な推奨値」をつくります。

### 第一段階：ルールベース

「水木は水木寿司を出す」「年金支給日は多め」「雨なら控えめ」など、現場のルール・経験則をそのままコード化したもの。

### 第二段階：過去平均ベース

曜日別・商品別・天気別の単純平均販売数。

### 第三段階：機械学習モデル

ランダムフォレストや LightGBM などの学習済みモデルによる予測。

### 並列表示の意義

機械学習の予測値**だけ**を出すのではなく、ルールベース・過去平均・モデル予測を**並べて**提示するのが核心です。

例：

```
過去平均：38 パック
ルール補正後：44 パック
モデル予測：41 パック
最終推奨：42 パック
```

「AI がそう言ってます」より、「過去平均では 38、年金支給日補正で 44、モデルは 41 なので、標準推奨は 42 です」の方が現場に通ります。寿司場だけでなく、行政提案にも通じる説明の型です。

---

## 2. 製造計画支援機能

販売数予測だけでは現場価値が薄いため、発注・人員配置までつなげます。これにより「販売予測アプリ」が「製造計画支援アプリ」に変わります。

出力すべき項目：

- シャリ必要量
- ネタ必要量
- パック必要数
- 作業時間目安
- 必要人員
- 開店前に作る数
- 昼前に追加する数

例：

```
明日の推奨製造数：72 パック
想定シャリ量：〇 kg
必要パック：72 枚＋予備 10 枚
開店前：50 パック
10:30 追加判断：22 パック
必要作業時間：2.5 時間
```

ここまで出せると、店長目線での価値が出ます。

## 3. 責任設計

実務導入する場合、「責任の置き方」も設計に含める必要があります。

アプリが出すのは**命令ではなく推奨値**。画面に明示します：

> この予測は製造判断を補助するものであり、最終判断は現場責任者が行います。

地味ですが重要です。外したときに「アプリが悪い」になると運用が死にます。**現場の経験値とデータを合わせるための道具**という位置づけにすることで、長く使われるツールになります。

---

## 4. 実装の優先順位

v2.0 で最初に取り組むべき改良は以下の 5 つ：

1. 商品カテゴリ別予測
2. 理由表示（なぜその数値になったかの根拠提示）
3. 通常日とイベント日の分離
4. 製造数・販売数・廃棄数の記録
5. 予測と実績の比較画面

この 5 つが入ると、実務アプリの体裁が整います。

---

## 5. ポジショニング

SNS や説明資料での 1 文：

> 今後は、単なる販売数予測ではなく、商品別の製造数、廃棄数、欠品リスク、イベント日補正まで含めた「現場の製造判断支援ツール」として改善していく予定です。

この方向で作ると、「AI で遊びました」から「現場業務をデータ化して改善する人」に見え方が変わります。寿司アプリが、ただのネタから本当にネタ場の参謀になります。
