# 手法解説

> 📝 **本ドキュメントについて**
> このドキュメントは Claude Code（Anthropic 社のコーディング支援 AI、Opus 4.7）が執筆し、プロジェクト管理者が監修したものです。後述するセクション 4「なぜLLMじゃなくていいのか」は、本プロジェクトの**予測タスク**に関する議論であり、開発支援用途での LLM 利用を否定するものではありません。むしろ本プロジェクト自体が「AIの適材適所」の一例です。

---

## 0. 全体像

本プロジェクトでは、寿司の販売数を予測するために 4 つのモデルを実装しました。それぞれ性質が大きく異なり、G検定の出題範囲を網羅するように選定されています。

| モデル | タスク | 学習手法 | 特徴 |
|---|---|---|---|
| 線形回帰 | 回帰 | 最小二乗法 | 解釈性が最高、ベースライン |
| ロジスティック回帰 | 分類 | 最尤推定（勾配降下）| 確率出力、解釈性が高い |
| ランダムフォレスト | 回帰 | バギング + 決定木 | 非線形パターン、特徴量重要度 |
| LightGBM（勾配ブースティング）| 回帰 | ブースティング + 決定木 | 高精度、業界標準 |

## 1. 4つのモデルの概要

### 1.1 線形回帰 (Linear Regression)

**ひとことで：** 特徴量に重みをかけて足し合わせ、連続値を予測する最も基本的な回帰モデル。

#### 数学的定式化

入力ベクトル `x = (x₁, x₂, ..., xₙ)` に対して、予測値 `ŷ` は次のように表現されます：

```
ŷ = β₀ + β₁·x₁ + β₂·x₂ + ... + βₙ·xₙ
```

ベクトル表記なら `ŷ = Xβ`（バイアス項を `x₀ = 1` として β に含める）。

学習では、訓練データの実測値 `y` と予測値 `ŷ` の二乗誤差の和を最小化する `β` を求めます：

```
β* = argmin Σ (yᵢ - ŷᵢ)²
       β    i
```

これが**最小二乗法（OLS：Ordinary Least Squares）**です。解析的に閉じた解 `β = (XᵀX)⁻¹Xᵀy` が存在するため、勾配降下のような反復計算は不要です。

#### 直感的な意味

「気温が 1℃ 上がると販売数は β（係数）個変化する」という形で、各特徴量の効き方を係数として読み取れます。本プロジェクトでは `StandardScaler` で特徴量を標準化してから学習しているため、係数の絶対値の大きさで特徴量の重要度を比較できます（特徴量重要度タブで可視化）。

#### 長所と短所

- **長所**：解釈性が抜群、学習・推論ともに高速、データが少なくても安定。
- **短所**：特徴量と目的変数の関係が線形でないと精度が頭打ち。外れ値に弱い（二乗誤差が大きく引っ張られるため）。

---

### 1.2 ロジスティック回帰 (Logistic Regression)

**ひとことで：** 線形回帰の出力を「確率」に変換し、クラスを予測する分類モデル。

#### なぜ「回帰」なのに分類？

歴史的経緯です。線形回帰の枠組み（線形結合 `Xβ`）を使って**ログオッズ（対数オッズ）を回帰する**ため、「ロジスティック回帰」と呼ばれます。出力はクラス確率なので用途は分類です。

#### 数学的定式化（2クラスの場合）

線形結合 `z = Xβ` をシグモイド関数で 0〜1 に押し込めることで確率にします：

```
σ(z) = 1 / (1 + e^(-z))
```

`σ(z) ≥ 0.5` ならクラス 1、それ未満ならクラス 0 と判定します。

#### 多クラス分類（本プロジェクト：3クラス）

本プロジェクトでは「少/普/多」の 3 クラスを予測するため、シグモイドの代わりに **ソフトマックス関数** を使います：

```
P(y = k | x) = e^(zₖ) / Σ e^(zⱼ)
                       j
```

各クラス `k` に対して線形結合 `zₖ = Xβₖ` を計算し、ソフトマックスで全クラスの確率の和が 1 になるよう正規化します。最も確率が高いクラスを予測値として採用します。

#### 本プロジェクトでの離散化

販売数（連続値）を、商品ごとの**33/66 パーセンタイル**で 3 クラスに切り分けています：

- 0：少（下位 33%）
- 1：普（中位 33%）
- 2：多（上位 33%）

> **退化ケース：** ランチ寿司（P004）は約半数が 0 個（販売日以外）のため、33 パーセンタイルと 66 パーセンタイルが重なります。`fit_sales_thresholds()` で「最小の正値」までフォールバックして順序を保証しています（`src/feature_engineering.py`）。

#### 長所と短所

- **長所**：クラス確率が出るので意思決定（しきい値調整）に使いやすい。係数で解釈可能。
- **短所**：特徴量と対数オッズの関係が線形であることを仮定する。複雑な決定境界は引けない。

---

### 1.3 ランダムフォレスト (Random Forest)

**ひとことで：** 決定木を多数（本プロジェクトでは 200 本）作って予測を平均する、**バギング（Bagging）**の代表手法。

#### 決定木の復習

決定木は「ある特徴量がしきい値以上か未満か」で分岐を作り、葉ノードで予測値を返すモデルです。

```
[is_sale_day == 1?]
   /            \
 No              Yes
 [base × 0]      [temperature ≥ 30?]
                  /            \
                No              Yes
                [base × 1.5]    [base × 0.8 × 1.5]
```

決定木は**非線形パターンを自然に表現**できますが、深く育てると過学習します。

#### バギングの仕組み

1. 訓練データから**ブートストラップサンプリング**（重複ありランダム抽出）で多数のサブセットを作る
2. 各サブセットで決定木を 1 本ずつ学習（特徴量もランダムにサブセットを選ぶ）
3. 予測時は全木の予測値を**平均**（回帰の場合）または**多数決**（分類の場合）する

これにより**バリアンス（予測のばらつき）を下げる**効果があります。1 本の木は不安定でも、大量に集めて平均すれば安定します。

#### なぜ本プロジェクトで効いたか

特に **P004（ランチ寿司）** で線形回帰の MAE 3.43 → ランダムフォレスト 2.16 と大幅に改善しました。これは「販売日以外は 0、販売日は通常通り」という**強い非線形パターン**を、決定木の分岐構造（`is_sale_day == 1?`）が直接的に表現できるためです。線形モデルでは One-Hot 化した `is_sale_day` の係数だけでは表現しきれません。

#### 長所と短所

- **長所**：非線形を扱える、特徴量重要度が自然に得られる、ハイパーパラメータに比較的鈍感。
- **短所**：解釈性が線形モデルより劣る（200 本の木の合議）、メモリ消費が多い、外挿（訓練データ範囲外）に弱い。

---

### 1.4 勾配ブースティング (Gradient Boosting / LightGBM)

**ひとことで：** 弱い予測器（浅い決定木）を順番に追加し、前の予測の誤差を埋めていく**ブースティング（Boosting）**手法。

#### バギングとの違い

| | バギング (RF) | ブースティング (GB) |
|---|---|---|
| 木の作り方 | 並列、独立 | 直列、前の木の誤差を学習 |
| 各木の深さ | 深い（過学習させる）| 浅い（弱学習器）|
| 効果 | バリアンスを下げる | バイアスを下げる |
| 並列化 | 容易 | 木単位では困難 |

#### アルゴリズム概略

`m` 番目の木 `f_m` は、それまでの予測 `F_{m-1}(x)` の**負の勾配**（≒誤差）を学習します：

```
F_m(x) = F_{m-1}(x) + η · f_m(x)
```

`η`（学習率、learning rate）で各木の寄与を調整し、過学習を抑えます。最終予測は全木の累積です。

#### LightGBM を採用した理由

- **高速**：ヒストグラムベースの分割探索、リーフワイズ成長で XGBoost より速い場合が多い
- **業界標準**：Kaggle や本番環境で広く使われている
- **依存軽量**：`pip install lightgbm` 一発、GPU なしでも実用的
- **Microsoft 製**：開発が活発、ドキュメントが整っている

#### 本プロジェクトでの結果

精度はランダムフォレストとほぼ同等（P004 で MAE 2.65 vs RF 2.16）。**チューニングなしのデフォルトパラメータ**で動かしているため、本来の性能を引き出せていません。M5 以降での `n_estimators` / `learning_rate` / `num_leaves` 等のグリッドサーチで改善余地があります。

#### 長所と短所

- **長所**：構造化データに対する精度はトップクラス、特徴量重要度が得られる。
- **短所**：ハイパーパラメータが多くチューニング前提、過学習しやすい、解釈性は RF より低い。

## 2. なぜこの4つを選んだか

G検定対策のポートフォリオとして、以下の **4 つの対比軸** をすべてカバーするよう選定しました：

### 2.1 線形 vs 非線形

- **線形側**：線形回帰、ロジスティック回帰
- **非線形側**：ランダムフォレスト、勾配ブースティング

特徴量と目的変数の関係を直線（超平面）で近似するか、それとも分岐構造で表現するか。本データでは P004 で非線形の優位性がはっきり見えました。

### 2.2 回帰 vs 分類

- **回帰**：線形回帰、ランダムフォレスト、LightGBM
- **分類**：ロジスティック回帰

同じ「販売数」というデータを、連続値として予測する（回帰）か、離散カテゴリに分類する（分類）かで使うモデルと評価指標が変わります。タスク設計の重要性を実感する仕掛けです。

### 2.3 バギング vs ブースティング

- **バギング**：ランダムフォレスト
- **ブースティング**：LightGBM

アンサンブル学習の二大潮流をどちらも実装。バリアンス削減 vs バイアス削減という設計思想の違いを比較できます。

### 2.4 解釈性 vs 精度

- **解釈性重視**：線形回帰（係数で寄与を読み取れる）、ロジスティック回帰
- **精度重視**：ランダムフォレスト、LightGBM

実務では「説明できる予測」が求められる場面（金融・医療・規制業界）と「とにかく当てたい」場面で使い分けます。

### まとめ：1つのデータセットで4つの設計判断軸を学べる

同じ寿司販売データに 4 モデルを適用することで、**「どの場面でどの手法が向くか」を自分の手で確かめられる**構成にしました。これがG検定の試験対策としても、実務スキルとしても価値があります。

## 3. 評価指標の選び方

「精度が高い」とひとことで言っても、何をもって高いとするかは指標次第です。タスク（回帰 / 分類）と目的（外れ値対応 / 不均衡データ対応）で使い分けます。

### 3.1 回帰タスクの指標（線形回帰、RF、GB）

#### MAE（Mean Absolute Error、平均絶対誤差）

```
MAE = (1/n) · Σ |yᵢ - ŷᵢ|
```

- **意味**：予測が平均で何個ズレるか。単位が販売数と同じなので直感的。
- **特徴**：外れ値に対して**頑健**（誤差を線形にしか効かせない）。
- **本プロジェクトでの使い所**：実用上のズレを把握するメイン指標。

#### RMSE（Root Mean Squared Error、二乗平均平方根誤差）

```
RMSE = √( (1/n) · Σ (yᵢ - ŷᵢ)² )
```

- **意味**：MAEと同じ単位だが、**大きな誤差を重く罰する**（二乗するため）。
- **特徴**：外れ値に敏感。`MAE ≤ RMSE` が常に成り立ち、両者の差が大きいほど誤差のばらつきが大きい。
- **本プロジェクトでの使い所**：たまの大ハズレを許容したくない場面で重視。

#### R²（決定係数、Coefficient of Determination）

```
R² = 1 - Σ(yᵢ - ŷᵢ)² / Σ(yᵢ - ȳ)²
```

- **意味**：「平均値を予測する」というベースラインに対する改善度。
  - `R² = 1`：完璧な予測
  - `R² = 0`：平均値を出すのと同じレベル
  - `R² < 0`：平均値より悪い（モデルが破綻している兆候）
- **特徴**：単位が無次元なので異なるデータ間で比較しやすい。
- **本プロジェクトでの使い所**：商品間でモデルの「効き方」を比較するのに便利。

### 3.2 分類タスクの指標（ロジスティック回帰）

#### 正解率（Accuracy）

```
Accuracy = (正しく分類された件数) / (全件数)
```

- 直感的だが、**クラス不均衡があると誤解を生む**。例：100 件中 95 件が「少」なら、全件「少」と予測するだけで Accuracy 95%。
- 本プロジェクトでは P004 が「少」に偏るため、Accuracy 91% が見かけ上高く出ます（ベースラインも高い）。

#### 適合率（Precision）と 再現率（Recall）

| 指標 | 定義 | 直感 |
|---|---|---|
| Precision | TP / (TP + FP) | 「多」と予測したもののうち、実際に「多」だった割合 |
| Recall | TP / (TP + FN) | 実際に「多」だったもののうち、「多」と予測できた割合 |

- **トレードオフ関係**：閾値を厳しくすると Precision が上がり Recall が下がる、逆も然り。
- 多クラスの場合は **macro 平均**（クラスごとに計算して単純平均）か **micro 平均**（全件まとめて計算）を選びます。本プロジェクトは macro 平均を採用（クラス不均衡で各クラスを公平に評価するため）。

#### F1 スコア

```
F1 = 2 · Precision · Recall / (Precision + Recall)
```

- 適合率と再現率の**調和平均**。両方が高くないと F1 は上がらない。
- バランス重視の総合指標として、Accuracy より信頼できる場合が多い。

#### 混同行列（Confusion Matrix）

| | 予測：少 | 予測：普 | 予測：多 |
|---|---|---|---|
| 実際：少 | TN | FP | FP |
| 実際：普 | FN | TN | FP |
| 実際：多 | FN | FN | TP |

「どの方向の間違いが多いか」を見るのに必須。本プロジェクトの Tab3 で商品別に表示しています。

### 3.3 時系列交差検証（TimeSeriesSplit）

通常の **K-Fold 交差検証** はデータをランダムに分割しますが、時系列データでこれをやると**未来のデータで学習して過去を予測する**というカンニングが発生します（リーク）。

`TimeSeriesSplit` は時間順を保ったまま、過去で学習 → 未来で評価を繰り返します：

```
Fold 1: [訓練 ###      ][評価 ##][.........]
Fold 2: [訓練 #####    ][評価 ##][.......]
Fold 3: [訓練 #######  ][評価 ##][.....]
Fold 4: [訓練 ######### ][評価 ##][...]
Fold 5: [訓練 ###########][評価 ##][.]
```

本プロジェクトでは **5-fold** で評価しています。販売予測は時系列タスクなので、これが**唯一正しい交差検証手法**です。

## 4. なぜLLMじゃなくていいのか

このセクションは本プロジェクトの主張の中核です。ただし**冒頭の透明性注記の通り**、議論の対象は「予測タスクに LLM を使う必要性」であり、「LLM 全般の有用性」ではありません。最後に「**AIの適材適所**」として両者を整理します。

### 4.1 タスクの性質

販売数予測は次の特徴を持ちます：

- **入力が構造化データ**：日付、天気、気温、降水量などの数値とカテゴリ
- **出力が連続値（または離散クラス）**：自然言語ではない
- **入力と出力の関係が比較的単純**：曜日効果、天気効果、特売効果などの加法的・乗算的な構造

LLM の本領は **非構造化データ（テキスト、画像、音声）の理解と生成**です。構造化データの数値予測には、表形式データに特化した手法（線形モデル、決定木系）の方が向いています。

### 4.2 データ量の壁

LLM は数十億〜数兆トークンで事前学習され、その知識を活用します。一方、本プロジェクトのデータは：

- **2,920 行 × 12 カラム**
- **訓練可能なサンプル数は商品ごとに 730 行**

この規模では、LLM の事前知識の出番がほぼありません。ファインチューニングしても 2,920 行では足りず、過学習します。**少量データには古典手法**が定石です。

### 4.3 解釈性の問題

LLM は数百億〜数兆のパラメータを持つブラックボックスです。「なぜこの日は 18 個と予測したのか」を**人間が読める形で説明できません**。

一方、線形回帰なら：

```
予測 = 12（基準）+ 5（水曜の係数）+ 3（晴れの係数）+ ...
```

と寄与を分解できます。ランダムフォレストや LightGBM でも `feature_importances_` で「どの特徴量が効いたか」を確認できます。**意思決定に使う予測には解釈性が必須**です（特に業務利用や規制対応）。

### 4.4 コストとレイテンシ

| | 古典手法（本プロジェクト）| LLM |
|---|---|---|
| 訓練コスト | 数秒（CPU で完結）| 数百〜数千ドル（GPU クラスタ）|
| 推論コスト | ミリ秒（無料）| API 1 リクエスト ≒ 数銭〜数十円 |
| レイテンシ | 即座 | 数秒（ストリーミング前提）|
| 依存 | scikit-learn / LightGBM | OpenAI / Anthropic 等のクラウド API |

販売予測のような「**頻繁に呼び出され、即答が求められる**」用途では、LLM はコスト・レイテンシの両面で過剰です。

### 4.5 再現性

古典手法は**乱数シード固定で完全再現**できます。本プロジェクトの `np.random.default_rng(42)`、`RANDOM_STATE=42` は、別マシンで同じバージョンの scikit-learn を使えば**バイト単位で同じ結果**を出します。

LLM はサンプリング温度 0 にしても再現性に揺らぎが生じる場合があり、モデル更新で挙動が変わります。**再現可能な実験には古典手法**が選ばれる理由です。

### 4.6 AIの適材適所——本プロジェクトのもう一つの結論

ここまでは「**予測タスクに LLM は不要**」という議論でした。しかし、本プロジェクト**そのもの**は **Claude Code（LLM）が書いた**ものです。これは矛盾でしょうか？

矛盾ではなく、**役割が違うだけ**です：

| 役割 | 適切な技術 | 理由 |
|---|---|---|
| 構造化データの数値予測 | 古典 ML（線形回帰・RF・GB）| データ量・解釈性・コスト |
| 非構造化な意図の翻訳（仕様 → コード）| LLM（Claude Code）| 自然言語処理が本領 |
| ドキュメント執筆・要約 | LLM | 文章生成が本領 |
| プロジェクト管理・意思決定 | 人間 | 価値判断・優先順位付け |

**「LLM を使うかどうか」は、タスクの性質で決める。** これがこのプロジェクトから得られる一番大きな教訓です。

## 5. 本プロジェクトの限界と今後

### 5.1 ダミーデータの限界

本プロジェクトは spec.md セクション 4 のルールに基づいて生成した**人工データ**を使っています。実データには次のような複雑さがあり、本プロジェクトでは扱っていません：

- **イベント効果**：地元の祭り、テレビ取材、SNS バズなどの単発要因
- **競合店の動向**：近隣の新店オープン、価格変更
- **マクロ経済**：景気、賃金、インバウンド需要
- **欠品・売り切れ**：上限が販売数に影響する非対称性

実データでこの 4 モデルを動かすには、これらを補助変数として取り込む特徴量設計が追加で必要です。

### 5.2 ハイパーパラメータ未チューニング

ランダムフォレストと LightGBM は `n_estimators=200` のみ指定し、他はデフォルトです。本来は `GridSearchCV` や `Optuna` で最適化すべきですが、M3 のスコープは「複数モデルを並べる」までとしました。チューニングで MAE が 10〜30% 改善する余地があると見込まれます。

### 5.3 信頼区間の簡易版

回帰モデルの信頼区間は「学習データ全体の残差標準偏差 σ × 1.96」という簡易計算です。**正確な予測区間**は statsmodels の `get_prediction()` などで、入力点ごとに分散を計算する必要があります。

### 5.4 特徴量重要度の表示

現状は「4 商品の平均」のみ表示。商品ごとの違い（例：P004 では `is_sale_day` が支配的）を見せるには、商品×特徴量のヒートマップが望ましいです。

### 5.5 今後の発展方向

1. **時系列モデルの追加**：ARIMA、Prophet、LSTM 等で「過去の販売トレンド」を直接モデリング
2. **多目的最適化**：販売数予測 → 仕入れ最適化（確率的最適化との接続）
3. **不確実性の可視化**：ベイズ手法、コンフォーマル予測で予測区間を厳密化
4. **本番化**：FastAPI で予測 API を切り出し、Streamlit はフロントエンドのみに。MLflow でモデル管理
5. **A/B テスト基盤**：複数モデルを同時運用し、業務 KPI で比較

これらは本プロジェクトのスコープ外ですが、ポートフォリオから業務応用に展開する自然な拡張です。
