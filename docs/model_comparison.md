## 1. 評価設定

### 1.1 データ

- **総レコード数**：4,380 行（2024-04-01 〜 2027-03-31、1,095 日 × 4 商品）
- **目的変数**：`sales_count`（販売数）
- **特徴量**：20 次元
  - 数値：`temperature`、`precipitation`、`month`、`effective_price`、`cpi_index`
  - フラグ：`is_weekend`、`is_holiday`、`is_pension_day`、`is_sale_day`
  - One-Hot：`weather`（4カテゴリ）、`day_of_week`（7カテゴリ）

### 1.2 分割と評価指標

- **交差検証**：5-fold `TimeSeriesSplit`（時間順を保ち、過去で学習 → 未来で評価）
- **回帰**：MAE、RMSE、R²
- **分類（3クラス：少/普/多）**：Accuracy、Precision (macro)、Recall (macro)、F1 (macro)、混同行列
- **離散化**：商品ごとの 33 / 66 パーセンタイルで「少 / 普 / 多」に切り分け（**v1.1 で fold ごとに訓練データのみで fit するよう修正済み**）

### 1.3 モデルと設定

| モデル | ライブラリ | 主な設定 |
|---|---|---|
| 線形回帰 | scikit-learn | `LinearRegression()` + `StandardScaler` |
| ロジスティック回帰 | scikit-learn | `LogisticRegression(max_iter=1000)` + `StandardScaler` |
| ランダムフォレスト | scikit-learn | `n_estimators=200` |
| LightGBM | LightGBM | `n_estimators=200`（他はデフォルト）|

すべて `random_state=42` 固定、`商品ごとに別モデル`を学習（合計 16 モデル）。

## 2. 回帰モデルの結果（v1.1）

### 2.1 商品 × モデル別 CV MAE と訓練 MAE

販売数の単位は「個」。**CV が小さいほど良い。CV と訓練の差が大きいほど過学習。**

| 商品 | 線形 (CV) | 線形 (訓練) | RF (CV) | RF (訓練) | LightGBM (CV) | LightGBM (訓練) |
|---|---:|---:|---:|---:|---:|---:|
| P001 水木寿司（10貫） | 5.491 | 2.805 | **1.614** | 0.503 | 2.098 | 0.895 |
| P002 春8貫寿司         | **1.802** | 1.510 | 1.794 | 0.593 | 1.999 | 0.824 |
| P003 絆10貫寿司        | 2.177 | 1.770 | **2.082** | 0.664 | 2.291 | 0.939 |
| P004 ランチ寿司         | 4.138 | 2.079 | **1.872** | 0.567 | 2.340 | 0.862 |

### 2.2 商品 × モデル別 CV R² と訓練 R²

**CV が 1 に近いほど良い。訓練と CV の R² ギャップが過学習の証拠。**

| 商品 | 線形 (CV) | 線形 (訓練) | RF (CV) | RF (訓練) | LightGBM (CV) | LightGBM (訓練) |
|---|---:|---:|---:|---:|---:|---:|
| P001 | 0.577 | 0.923 | **0.932** | 0.994 | 0.928 | 0.988 |
| P002 | 0.707 | 0.802 | 0.685 | 0.969 | 0.611 | 0.935 |
| P003 | 0.720 | 0.820 | 0.715 | 0.972 | 0.649 | 0.937 |
| P004 | **0.190** | 0.899 | **0.879** | 0.989 | 0.841 | 0.977 |

**重要な観察**：
- **線形回帰 P004 の CV R² = 0.19**：壊滅的。後述の通り、商品の販売スケジュール（水木以外）と `is_sale_day`（水曜・週末）が非自明な交互作用を持ち、線形モデルでは表現できない。
- **RF/GB の訓練 R² がほぼ 1.0**：訓練データに対しては完璧に近いが、CV では大きく落ちる → 過学習。ハイパラチューニングで改善余地あり。
- **線形回帰の訓練 R² と CV R² のギャップは比較的小さい**：過学習しにくい（モデル自由度が低い）。これは線形モデルの長所。

## 3. 分類モデルの結果（ロジスティック回帰、v1.1）

### 3.1 全体指標（CV）

| 商品 | Accuracy | Precision (macro) | Recall (macro) | F1 (macro) |
|---|---:|---:|---:|---:|
| P001 | **0.992** | 0.791 | 0.800 | 0.795 |
| P002 | 0.678 | 0.652 | 0.648 | 0.643 |
| P003 | 0.664 | 0.600 | 0.655 | 0.607 |
| P004 | 0.825 | 0.841 | 0.822 | 0.807 |

P001 の Accuracy 99.2% は、**水木以外は確定で 0（=「少」）**という強いルールで Accuracy が押し上げられているため。F1 (macro) は 0.795 と Accuracy より控えめ。「水木のうち、どれが普 / 多か」の判別が本質的な難しさ。

### 3.2 P004 の Accuracy が高い理由

ランチ寿司は「販売日以外は 0」という強いルールがあり、約半数が「少（販売数 0）」クラスに固定されます。このベースラインだけで Accuracy が 50% を超えるため、見かけ上の数字が高く出ます。**F1 (macro) で評価すると 0.819 と他商品を上回りますが、Accuracy ほどの差はない**点に注意してください。

### 3.3 混同行列（参考：実際のアプリの Tab3 で各商品を切り替え可能）

例として P002（8貫寿司）の混同行列（5-fold 合計）を示すと、対角成分（正解）が支配的で、かつ「少 ↔ 多」のような大きな誤分類は少ない傾向にあります（隣接クラス間の取り違えが主）。これはロジスティック回帰が**順序を意識しないモデル**であるにも関わらず、特徴量から自然と順序関係を学習している証拠です。

## 4. 商品ごとの傾向と考察

### 4.1 P001〜P003（通常販売商品）

線形回帰、RF、GB の差は小さく、MAE で 0.5 個程度の差に収まります。R² も 0.62〜0.76 のレンジ。

**示唆**：これらの商品は「曜日効果 × 天気効果 × 祝日効果」が**ほぼ加法的（線形）に重なる**設計のため、線形回帰だけで十分に近い精度が出ます。アンサンブル手法の恩恵が薄い領域です。

### 4.2 P004（ランチ寿司、特売日のみ販売）

線形回帰の MAE 3.43 → **ランダムフォレスト 2.16** と劇的に改善（-37%）。LightGBM も MAE 2.65 と健闘。

**示唆**：「`is_sale_day == 0` ならば常に 0、それ以外は通常通り」という**強い非線形パターン**は、決定木の分岐構造で自然に表現できます。線形回帰では One-Hot 化された `is_sale_day` の係数が `× 20` 程度の効果しか持てず、販売日と非販売日の振る舞いの違いを完全には吸収できません。

これは G検定でよく扱われる「**線形モデルの限界**」と「**決定木系の利点**」の教科書的な例になっています。

## 5. なぜ複雑なモデルが線形回帰を圧倒しないのか

通常、ランダムフォレストや勾配ブースティングは線形回帰より精度が高いと期待されます。本データでは P004 を除き、その差は微小です。理由を整理します。

### 5.1 特徴量エンジニアリングが既に強い

本プロジェクトの 18 次元特徴量には、人手で設計した強力な情報が含まれています：

- `is_sale_day`（水曜・週末フラグ）
- `is_pension_day`（年金支給日フラグ）
- `is_holiday`（祝日フラグ）
- `month`（月）+ `dow_*` の One-Hot（季節性・曜日性）
- `temperature`、`precipitation`（連続値の天気情報）

これらは生成ロジック（`data/generate_data.py`）の倍率ルールにほぼ 1 対 1 対応しており、**線形モデルでも十分な情報量**になっています。RF や GB が新たに学べる「複雑なパターン」が少ないわけです。

### 5.2 データ規模が比較的小さい

商品ごとに 730 行という規模では、200 本の決定木をフィットさせる**自由度が過剰**で、過学習を起こしやすくなります。RF はバギングで耐性がありますが、それでも「全データから学べる本質的な情報」には限界があります。

### 5.3 ハイパーパラメータ未調整

LightGBM は `n_estimators` 以外デフォルトで動かしているため、本来の性能を引き出せていません。`learning_rate`、`num_leaves`、`min_child_samples` 等のチューニングで RF を超える可能性があります（M5 以降の課題）。

### 5.4 結論

「**複雑なモデルが常に勝つわけではない**」というのは、データサイエンスの実務で繰り返し確認される事実です。特徴量設計が良ければ線形モデルで十分、データが少なければ複雑モデルは過学習する。**モデルの選択はデータと特徴量の文脈に依存します**。

ただし v1.1 で**逆の現象**も観察できました：**P001 / P004 のようなスケジュール非線形性が強いデータでは、線形回帰が崩壊し（P004 R²=0.19）、RF が圧勝**する。データの構造が変われば、推奨モデルも変わる。これが**「データに向き合う」ことの実例**です。

## 6. 推奨と結論

| 用途 | 推奨モデル | 理由 |
|---|---|---|
| 解釈性が必要、または初期分析 | **線形回帰** | 係数で寄与を読み取れる、計算も速い |
| ゼロ膨張データや強い非線形がある | **ランダムフォレスト** | P004 のような場面で堅実に効く |
| 最高精度を狙う、チューニング前提 | **LightGBM** | デフォルトでも RF と同等、調整で伸びしろあり |
| クラス確率が必要、リスク管理用途 | **ロジスティック回帰** | 確率出力、しきい値調整可能 |

本プロジェクトの予測タスクとしては、**「線形回帰でベースライン、P004 だけランダムフォレストに切り替え」** という運用が実用上のスイートスポットです。コードの簡潔さ、推論速度、解釈性の総合点が最も高くなります。

## 7. 結果の再現方法

すべての数値は乱数シード固定で再現可能です。

```bash
# 1. 環境準備（README.md のセットアップ手順に準拠）
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# 2. データ生成（既に sushi_sales.csv がある場合はスキップ可）
python data/generate_data.py

# 3. 全 4 モデル × 4 商品 = 16 モデルを学習
python -m src.trainer
```

`src/trainer.py` の `main()` が、本ドキュメントの表 2.1 / 2.2 / 3.1 と同じ数値を標準出力に書き出します。差異が出る場合は scikit-learn と LightGBM のバージョンを確認してください（本プロジェクトの実行環境は scikit-learn 1.8.0、LightGBM 4.6.0）。

### D. 教訓

1. **「CV をやっていない」と「CV の中にリークがある」は別問題**：M5 時点では CV は実装されていたが、しきい値リーク・残差σバイアスという別の欠陥があった。
2. **データ構造を変えると、強かったモデルが崩れることがある**：P004 線形回帰の R² 崩壊が代表例。
3. **訓練と CV の両方を見ることで過学習が可視化される**：RF/GB の訓練 R² が 0.97-0.99 と CV を大きく上回ることが判明。チューニングよりも**評価の正確性**を先に確保すべきだった。
4. **データに向き合うこと**：アルゴリズムを変える前に、データの構造を実運用に合わせる作業の方が、改善幅は大きかった。
